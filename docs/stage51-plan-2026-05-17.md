## Deposit-order analysis (CRITICAL)

For self-host G3..G4 byte-identity to hold:
- `propagate_adj_multi` walks the body in same DFS order as
  single-walk `propagate_adj`.
- For param 0, deposits into `bucket[0]` arrive in identical
  order. `bucket_array_sum` reduction folds left-to-right.
- **Constraint: AST node positions in arena MUST be unchanged
  for param-0 path.** Verified by trace: `bucket_array_reset`
  and `set_param_array_name` write only to scratch slots
  (sb+90..105), not arena. Inside `differentiate_reverse_all`,
  `mk_node` calls fire in identical DFS order. Param-1+
  deposits don't shift param-0 arena indices.

**Byte-identity is preserved** for the n=1 case (which is what
the cascade exercises) — the existing Inc 2 bridge already
proved this. Stage 51 extends to n>1 only changes behavior
for `grad_rev_all` with multiple params (currently dogfood_05
binary_classifier with n=2).

## 6 regression tests needed

1. **Same-loss 2-param run** — `f(x, y)` with `[f.dx, f.dy]`
   pending. Both grad fns emitted, byte-identical to separate-
   walk output.
2. **Same-loss max-run (8 params)** — exercises bucket cap.
   No overflow trap (89001).
3. **Mixed-loss interleaving** — `[f.dx, g.dx, f.dy]`. All 3
   single-entry runs; run-detection stops at loss boundary.
4. **Single-entry run (n=1 identity)** — must produce byte-
   identical output to Inc 2 bridge path. Guards n=1
   regression.
5. **Invalid-field entry in run** — entry 1 has bad field name.
   Trap 88002 for that entry; entry 0 still produces valid
   gradient fn (fallback path).
6. **Chain of `grad_rev_all` across multiple loss fns** —
   `grad_rev_all(loss_a, loss_b)` with 2 params each in
   interleaved pending order. Tests two consecutive groups +
   tail advancement.

## Cascading-defect risk areas (4 hypotheses to probe in audit gate)

1. **`run_end` advancement off-by-one** — easy bug; entries
   silently skipped. Test 3 catches.
2. **Stale `field_l` reuse inside inner loop** — group path
   reads `field_l` per entry but easy to accidentally hoist.
   Corrupts mangled-name length for entries 1+.
3. **`have_param` hoisting** — must be per-entry; hoisting
   bypasses trap 88003 for invalid-field entries 1+.
4. **Poison-to-all-buckets in `propagate_adj_multi`** —
   currently appends only to bucket 0. For n>1, param 1+'s
   gradient silently emits 0.0 on unsupported node. Targeted
   fix in `propagate_adj_multi` itself.
