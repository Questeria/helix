# Stage 50 plan — bootstrap `grad_rev_all` single-walk port

## Stage Goal

Port the single-walk reverse-mode AD algorithm from the Python
side (`helixc/frontend/autodiff_reverse.py` — DONE at Stage 36)
to the in-Helix bootstrap compiler (`helixc/bootstrap/parser.hx`).
This closes the bootstrap side of **Tier 1 #3 — multi-output
reverse-mode AD**.

Beginner meaning: today the in-Helix compiler walks the loss
body once per gradient parameter (N walks for N params).
Mathematically correct, just slower. The Python compiler uses
a single walk that deposits adjoint contributions into N
parallel buckets, one per param, in O(body-size) work total
instead of O(N * body-size). This stage ports that algorithm
to the bootstrap compiler so the in-Helix toolchain is
algorithmically equivalent to the Python toolchain.

## Current state (pre-Stage-50)

`helixc/bootstrap/parser.hx` line 5402 documents the N-walk:

> This implementation walks once per target param (so for
> grad_rev_all with N params, we walk N times); a future
> optimization could simultaneously deposit into N buckets in
> one walk.

The Stage 49 closure cleared the runtime-tag prep work, so
Stage 50 starts on a clean working tree.

Touchpoints:
- `bucket_head_slot` / `bucket_count_slot` (sb+72/73): single
  bucket. Stage 50 adds N-parallel buckets.
- `propagate_adj(sb, node, adj, var_s, var_l)`: matches against
  ONE param. Stage 50 adds a multi-param variant.
- `differentiate_reverse_one`: seed adj=1.0_f64, walk once for
  ONE param. Stage 50 adds `differentiate_reverse_all` walking
  once for N params.
- Caller loop at parser.hx:5987-6116 iterates pending entries
  one at a time. Stage 50 refactors to batch entries by loss
  name and do one walk per loss.

## Increment breakdown

### Inc 1 — Multi-bucket scratch infrastructure (groundwork)

Add new scratch slots (sb+90 .. sb+121: 32 slots for up to 8
param-names + 8 bucket-heads + 8 bucket-counts + 8 reserved):

- `param_n` slot (sb+88): count of currently-active params (0-8)
- Param names at `sb+90 .. sb+105` (8 pairs of (start, len))
- Bucket heads at `sb+106 .. sb+113` (8 heads)
- Bucket counts at `sb+114 .. sb+121` (8 counts)

New helpers:
- `bucket_array_reset(sb, n)` — sets active n, zeros all
- `bucket_array_set_param(sb, idx, s, l)` — records a param name
- `bucket_array_append(sb, idx, expr)` — append to bucket idx
- `bucket_array_sum(sb, idx) -> i32` — sum bucket idx into expr
- `param_idx_of(sb, var_s, var_l) -> i32` — linear scan, return
  matching idx (0..n-1) or -1
- `propagate_adj_multi(sb, node, adj)` — like propagate_adj but
  scans against all active params and deposits into matching
  bucket
- `differentiate_reverse_all(sb, expr_idx) -> i32` — walks once
  with seed=1.0_f64. Returns 0; caller reads buckets via
  bucket_array_sum.

OLD helpers (bucket_head_slot, propagate_adj,
differentiate_reverse_one) stay UNCHANGED — Inc 1 only adds
new infrastructure. The caller still uses the old N-walk path.
This keeps the self-host cascade byte-identical at G3..G4
(since the actual code path executed at G2 hasn't changed).

### Inc 2 — Refactor the caller loop to use the new infrastructure

Group pending entries by loss-name. For each group:
1. Inline the loss body ONCE (cache the result).
2. Initialize bucket_array with the param names from each
   pending entry's `.d<param>` field.
3. Call `differentiate_reverse_all` ONCE.
4. For each pending entry in the group, read its bucket via
   bucket_array_sum, simplify, and emit the synthesized
   gradient fn.

Old `differentiate_reverse_one` stays for back-compat / fallback,
but the production path now uses the new helpers.

### Inc 3 — Stage 50 Closure (3/3 clean gates)

Same protocol as Stage 35-49. Expected cascading-defect rhythm
(per Stage 46/48/49 precedent) — at least one HIGH per gate
until convergence.

## Self-host invariant

The change MUST preserve `test_selfhost_cascade` G3..G4
byte-identical. Inc 1 is purely additive (new code never
called at G2), so cascade is trivially preserved. Inc 2 changes
the executed code path; the algorithmic output (final AST shape)
must be identical to the N-walk version modulo bucket-ordering.

Risk: if the bucket-deposit ORDER differs from the N-walk
deposit order, the synthesized derivative AST might have a
different shape (sum-chain ordered differently). This produces
mathematically-equivalent but byte-different output. To avoid
breaking G3..G4 invariance, Inc 2 must preserve the same
deposit order as the N-walk path.

Mitigation: in `propagate_adj_multi`, when matching against
param names, iterate in REGISTRATION ORDER (same as the
pending-entry order in the caller loop). Each AST_VAR match
deposits into exactly one bucket. The walk order is the same
top-down DFS, so the deposit sequence within each bucket
matches the N-walk version.
