# Stage 49 plan — runtime Ok/Err tag for Result<T,E>

Drafted 2026-05-17 while Stage 48 closure audits run in parallel.
Picks up directly after Stage 48 declares CLOSED.

## What Stage 49 ships

Per the roadmap (docs/ROADMAP.md:81-84): Tier 4 #14 Inc 3 — the
**runtime Ok/Err tag** that gives Result<T, E> real runtime
discrimination. Unlocks the 4 currently-typecheck-rejected
builtins:

- `is_ok(r)` / `is_err(r)` — currently rejected (Stage 46 closure
  gate-1 F1; in typecheck.py search for
  `if bn in ("is_ok", "is_err"):`) because no tag exists.
- `map_err(r, new_err)` — currently rejected (Stage 46 F2; in
  typecheck.py search for `if bn == "map_err":`) for the same
  reason.
- `unwrap_ok` / `unwrap_err` on dynamic Result (call-returning) —
  currently we panic-or-undefined at runtime because the wrong-arm
  is undetectable. With a tag we can panic deterministically.

Plus the **real `?` propagation branch**: today `__try(r)` is
identity-lowered (Stage 48 closure gate-1 ledger:1-end). Once
the tag exists, `__try` becomes a real conditional-branch IR
that early-returns the Err up the call stack — the canonical
Rust/Swift `?` semantics.

## Why this is multi-touchpoint (= scope carefully)

Stage 46-48 audited closures consistently caught silent
miscompiles where one site assumed identity-lowering and a
sibling site assumed runtime discrimination. The Stage 46
3-layer wrong-arm safety net (TyUnknown hint + per-binding
provenance map + Assign-arm invalidation + non-constructor-RHS
pop) currently masks the missing runtime tag for **statically-
constructed** Results. Stage 49 must:

1. Keep the static-provenance pathway intact (don't regress
   wrong-arm detection for `let r: ... = Ok(5); unwrap_err(r)`).
2. Add a runtime tag that activates whenever the static
   pathway can't decide (e.g. call return values, branches
   that produce different arms, params of Result type).
3. Reroute the 3 currently-rejected typecheck arms to allow
   the call AND emit the appropriate IR.
4. Preserve `?` Phase-0 dogfood (`dogfood_17_try_operator.hx`
   still exits 42) — the static-Ok pathway still works without
   tag overhead.

## Proposed memory layout

Two-slot Result representation:

```
struct Result<T, E> {
    i32 tag;       // 0 = Ok, 1 = Err
    union {
        T ok;
        E err;
    } payload;
}
```

Open design questions:
- **Tag width**: 1 byte (compact) vs 4 byte (aligned, simpler).
  Recommendation: **4-byte tag** + 4-byte (or larger) payload,
  giving 8 bytes minimum. Aligns with current SysV register
  pair return (rax + rdx) — tag in rax, payload in rdx.
- **Tag-0 vs tag-1 for Ok**: convention is "0 = success" in
  POSIX, so Ok=0, Err=1. Branches read more naturally
  (`if tag == 0 { unwrap_ok } else { propagate }`).
- **Wider payloads**: `Result<i64, i64>` needs 8-byte payload.
  `Result<f64, f64>` likewise. `Result<i32, bool>` — payload
  width = max(sizeof(T), sizeof(E))? Or per-arm?

Settled answer for Inc 1: tag = i32 (4 bytes), payload = i32
(4 bytes). Both i32 because that covers our current dogfood
set. Wider payloads = Inc 4+ (or a separate stage).

## Calling convention impact

A function `fn safe_div(...) -> Result<i32, i32>` today returns
a single i32 (the identity-lowered Ok inner). Post-Stage-49 it
returns 8 bytes total = tag (4) + payload (4).

SysV options:
- **Option A**: pair-register return (`rax` + `rdx`). Both
  caller and callee learn the convention. Cleanest.
- **Option B**: pack into rax as a 64-bit value (tag in high
  32, payload in low 32 — or vice versa). Avoids touching
  multi-register handling. Simpler.

Recommendation: **Option B** for Inc 1 (smaller backend change),
Option A for Inc 4+ (when wider payloads need it).

## Increment breakdown (3-clean-gate per Inc, same as Stage 46-48)

### Inc 1 — TIR opcode + Ok/Err construction with packed tag

- New TIR ops: `RESULT_PACK(tag, payload) -> packed_i64`,
  `RESULT_TAG(packed) -> i32`, `RESULT_PAYLOAD(packed) -> i32`.
- `Ok(v)` lowering: `RESULT_PACK(0, v)`.
- `Err(e)` lowering: `RESULT_PACK(1, e)`.
- `unwrap_ok` lowering: `RESULT_PAYLOAD(r)` + runtime check
  `if RESULT_TAG(r) != 0 { halt(exit_code=2) }`.
- `unwrap_err` lowering: `RESULT_PAYLOAD(r)` + runtime check
  `if RESULT_TAG(r) != 1 { halt(exit_code=2) }`.
- Result-typed fn return type lowers to i64 (packed).
- All existing dogfood programs must still pass (static-Ok
  pathway means tag is always 0, payload extraction unchanged).

### Inc 2 — Enable is_ok / is_err

- Drop the Stage 46 F1 typecheck rejection.
- `is_ok` lowering: `RESULT_TAG(r) == 0`.
- `is_err` lowering: `RESULT_TAG(r) != 0`.
- New test: dynamic Result through a call returns the right
  tag, `is_err(safe_div(10, 0)) == true`.

### Inc 3 — Enable map_err

- Drop the Stage 46 F2 typecheck rejection.
- `map_err(r, new_err)` lowering: branch on tag — if Err, repack
  with `new_err`; if Ok, pass through unchanged.
- New test: `unwrap_err(map_err(Err(5), 99)) == 99`.

### Inc 4 — Real `?` propagation branch

- Change `__try` IR lowering from identity-pass-through to
  conditional early-return.
- `__try(r)` lowering:
  ```
  if RESULT_TAG(r) == 1 {
      return r;  // propagate Err
  }
  RESULT_PAYLOAD(r)  // extract Ok
  ```
- Phase-0 dogfood_17 still exits 42 (both calls produce Ok).
- New test: `let r: Result<i32, i32> = safe_div(10, 0)?; ...`
  where caller returns Result — Err propagates correctly.

### Inc 5 — Closure (3/3 clean audit gates)

Same protocol as Stage 35-48. Expect at least 1 silent-failure
finding because this is a multi-touchpoint change.

## Open risks

1. **Self-host cascade**: every G2..G4 stage compiles a Helix
   program that constructs Result values. The packed-tag
   representation must be byte-identical across G2/G3/G4 OR
   the cascade breaks. **Mitigation**: pin to Inc 1 small
   change first, run cascade after EACH Inc.
2. **Memory wrappers**: `Known<Result<i32, i32>>` —
   the outer wrapper is identity-lowered (Phase-0), but if
   the inner Result is now 8 bytes packed, the wrapper's
   `unwrap` becomes a 64-bit read. Need to update wrapper
   accessor sizes.
3. **f64 inside Result**: `Result<f64, i32>` — payload is f64,
   needs 8 bytes alone, plus tag → 12+. Out of scope for Inc 1.
4. **Pointer payloads**: not in scope (no pointer/heap types
   in current Helix surface).

## Out-of-scope (Stage 50+)

- Wide payloads (i64, f64, struct).
- Auto-promotion to panic at top-level if Result is unhandled.
- `?` for Option<T> (no Option type yet).
- `Result<T, E>` with `T` or `E` themselves wrapper types
  beyond depth-1 (Known<Result> handled; Known<Result<Known<i32>, ...>>
  may need wrapper-aware sizing).
