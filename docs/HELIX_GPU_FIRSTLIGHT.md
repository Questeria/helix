# Helix GPU first-light — kovc-emitted PTX executes on real hardware

**Date:** 2026-06-01 · **Milestone:** the Helix v1.0 DoD criterion-#3 GATE *mechanism* is demonstrated — Helix drives a real GPU end-to-end. **Independently audited PASS (2 adversarial agents, HIGH confidence, with negative controls).**

---

## The claim (precisely scoped)

`kovc` — the Helix self-hosting compiler, itself minted from the from-raw-binary seed (`hex0` → … → `seed.c` → `kovc`), with **no Python compiler and no nvcc in its own build** — compiled a `vector_add` `@kernel` written in Helix into PTX, and that **kovc-emitted PTX executed on the RTX 3070 Laptop GPU and produced numerically correct results** (`c[i]=a[i]+b[i]`; `c[7]=21` verified over all 256 elements).

This is **first-light for the elementwise class.** It does **not** by itself claim all of criterion #3 (the full tiled-GEMM / transformer-op corpus on hardware) — that breadth is phase P5 of `HELIX_FINISH_PLAN.md`. What it proves is the load-bearing thing everyone said was the single biggest blocker: **the GATE mechanism works — a raw-binary-bootstrapped Helix toolchain can emit PTX that runs correctly on a real GPU.**

---

## The pipeline (zero Python; one C launcher as a trusted-tool boundary)

```
helixc/examples/vector_add_kernel.hx   (a concrete @kernel, Helix source)
        |  kovc (minted by the raw-binary seed; emit_auto_for_ast_to_path routes @kernel -> PTX)
        v
/tmp/out.ptx   (945 bytes, kovc-emitted PTX text)
        |  helixc/runtime/cuda_launch.c  (CUDA Driver API: cuModuleLoadData + cuLaunchKernel)
        v
RTX 3070 Laptop GPU   ->   c[7]=21 (want 21) PASS
```

The `cuda_launch.c` C shim links the dynamic `libcuda.so`, which the static syscall-only Helix ELF cannot do today; it is an explicit **trusted-tool boundary** outside the self-host fixpoint (like `ptxas` / the build ladder — Decision D2). In-Helix dynamic linking retires it before the formal v1.0 freeze. The PTX *emission* — the part that proves "Helix can target the GPU" — is entirely kovc's.

## The kovc-emitted PTX (verbatim)

```ptx
.version 8.0
.target sm_75
.address_size 64

.visible .entry vector_add(.param .b64 param_0, .param .b64 param_1, .param .b64 param_2, .param .b64 param_3)
{
    .reg .pred  %p<256>;
    .reg .b32   %r<256>;
    .reg .b64   %rd<256>;
    .reg .f32   %f<256>;
    .reg .b16   %h<256>;

    mov.u32 %r0, %ctaid.x;
    mov.u32 %r1, %ntid.x;
    mul.lo.s32 %r2, %r0, %r1;
    mov.u32 %r3, %tid.x;
    add.s32 %r4, %r2, %r3;
    ld.param.u64 %rd0, [param_0];
    cvta.to.global.u64 %rd1, %rd0;
    mul.wide.s32 %rd2, %r4, 4;
    add.s64 %rd3, %rd1, %rd2;
    ld.global.f32 %f0, [%rd3];
    ld.param.u64 %rd4, [param_1];
    cvta.to.global.u64 %rd5, %rd4;
    mul.wide.s32 %rd6, %r4, 4;
    add.s64 %rd7, %rd5, %rd6;
    ld.global.f32 %f1, [%rd7];
    add.f32 %f2, %f0, %f1;
    ld.param.u64 %rd8, [param_2];
    cvta.to.global.u64 %rd9, %rd8;
    mul.wide.s32 %rd10, %r4, 4;
    add.s64 %rd11, %rd9, %rd10;
    st.global.f32 [%rd11], %f2;
    ret;
}
```

This is unmistakably kovc's own codegen (distinct from nvcc): hardcoded `.target sm_75` (the box is native sm_86; nvcc would emit sm_86), fixed over-allocated `%r<256>/%rd<256>/%f<256>` register files, and the address computation (`ld.param → cvta.to.global → mul.wide.s32 ·,4 → add.s64`) re-emitted for **every** array access with no common-subexpression elimination. nvcc emits a `// Generated by NVIDIA NVVM Compiler` banner, minimal registers, and one hoisted index — none of which appear here.

## Reproduce from the repo (exact)

```bash
# 1. assemble the PTX-driver source (frozen lexer+parser+kovc lib + a PTX-driver main)
python stage0/helixc-bootstrap/assemble_k1.py            # writes stage0/helixc-bootstrap/k1ptxdrv.hx

# 2. mint the PTX-driver kovc with the raw-binary seed (~10 min; 587093 bytes). NOTE the argv form:
wsl -e bash -c "cd /mnt/c/Projects/Kovostov-Native/stage0/helixc-bootstrap && ulimit -s unlimited && ./seed.bin k1ptxdrv.hx _kovc_ptx_driver.bin"

# 3. emit PTX from the Helix kernel + launch on the GPU (ONE wsl session -- /tmp is volatile across sessions)
wsl -e bash -c "cd /mnt/c/Projects/Kovostov-Native && cp helixc/examples/vector_add_kernel.hx /tmp/kernel_in.hx && stage0/helixc-bootstrap/_kovc_ptx_driver.bin; cat /tmp/out.ptx; gcc helixc/runtime/cuda_launch.c -I/usr/local/cuda/include -L/usr/lib/wsl/lib -lcuda -o /tmp/cl && /tmp/cl /tmp/out.ptx vector_add 256"
# expect: GPU [NVIDIA GeForce RTX 3070 Laptop GPU] kernel 'vector_add' over 256 elems: c[7]=21 (want 21) -> PASS
```

Notes: the driver's exit code is the emitted byte-count mod 256 (e.g. `177` for the 945-byte PTX) — that is success, not an error. The minted `_kovc_ptx_driver.bin` is git-ignored (a build artifact); regenerate it from the seed via step 2.

## Independent adversarial audit (2026-06-01)

Two fresh-context adversarial agents were tasked to **break** the claim. Both reproduced the chain on hardware and returned **PASS / HIGH**:

- **Critic — PASS, p_pass 0.95, HIGH.** Re-ran the driver → byte-identical 945-byte PTX (sha256 `f29d2ba7…`). Decisive **negative control**: mutating the emitted `add.f32`→`mul.f32` made the GPU return `c[7]=98` (=7×14) and the launcher correctly FAIL — proving the GPU JIT-executes the PTX *text*, the PASS is not hardcoded.
- **Skeptic — PASS, p_pass 0.90, HIGH.** nvcc fingerprint contrast (no NVVM banner, `%r<256>` regfile, 3× redundant addressing vs nvcc's 1×) proves the PTX is kovc's, not smuggled nvcc. Three more negative controls (sub→FAIL, no-op→`c[7]=0` FAIL, wrong-name→`cuModuleGetFunction` error) prove the launcher's verification is not hollow. `readelf`: the driver is a static, syscall-only, single-PT_LOAD ELF with **zero** CUDA/nvcc/ptxas strings; `cuda_launch.c` has no `system()`/`popen()`/shell-out.

Stricter-verdict-wins: **PASS, HIGH, p_pass 0.90, evidence reproducible.**

## GPU compute — naive matmul (Step B, 2026-06-01)

**The first real GPU *compute* kernel in pure Helix.** `kovc` compiled
`helixc/examples/naive_matmul_kernel.hx` — a naive matrix multiply (one thread per
output cell: `row=block_idx()`, `col=thread_idx()`; an f32 accumulator k-loop summing
`A[row,k]*B[k,col]`; reading `K`/`N` as scalar params) — to PTX, and it **executed on the
RTX 3070 computing a 16×16×16 matmul whose all 256 cells match a CPU reference (0 bad)**.
Committed `a38a1b6`. Pipeline + repro: as above, with
`cuda_launch out.ptx naive_matmul <N> matmul M K N` (the launcher allocates A/B/C,
launches `gridDim.x=M, blockDim.x=N`, and verifies every `M*N` cell in the kernel's
accumulation order over integer-valued inputs so f32 sums are exact).

This is the first kernel with a **loop + f32 accumulator + multi-statement body**, so it
surfaced (and we fixed + gated) three real `kovc.hx` PTX-emitter gaps — each invisible to
the elementwise kernels:
- **`AST_LET_MUT` (tag 12, `let mut`)** wasn't dispatched in `emit_ptx_expr` → the
  accumulator was silently dropped. Fixed via `ptx_is_let_tag`.
- **`AST_SEQ` (tag 13, statement sequences)** wasn't dispatched → a multi-statement body
  lost everything after the first statement. Added a `tag==13` arm.
- **Variable type tracking** — the vtab stored each var's register but not its type, so an
  f32 var read back as i32 (`add.s32`/`st.global.u32`). Added a 4th vtab slot `is_f`
  (4-wide entries, cap 12), `ptx_vtab_lookup_isf`, restore `vtab+55` on read, store on bind;
  plus `emit_ptx_assign` now branches `mov.f32` vs `mov.s32`.

**Independent adversarial audit (2 fresh agents) — PASS / HIGH:**
- **Critic p_pass 0.94** — reproduced; deterministic PTX (sha256 `c23547cb…`); negative
  controls flip (`add.f32`→`sub.f32` → 256 bad FAIL; corrupt stride → 144 bad FAIL; pristine
  → 0 bad PASS); multi-size 8/16/32 PASS; fixpoint IDENTICAL.
- **Skeptic p_pass 0.96** — **re-minted the driver byte-identically from the seed** (proving
  it's genuinely seed-minted); `readelf` static syscall-only ELF, zero nvcc/python strings;
  three negative controls all FAIL; sizes 8/16/32 + non-square 7×11×5 PASS; rebuilt the
  fixpoint K2==K3 with matching SHA-256.

**Gate:** matmul 256/256 on HW; elementwise regression green; **self-host fixpoint K2==K3
byte-identical** (595376==595376) — all edits PTX-path-only, the ELF self-host untouched.
**Scope:** correctness, not throughput — the naive matmul is the compute *floor*; a tiled
GEMM with shared memory (Step C) is the performance path.

## What remains for full criterion #3 (P5)

First-light is the elementwise floor. Full "GPU executes" needs the **throughput + breadth** corpus on hardware: a general tiled matmul (shared-memory staging + `bar.sync`, not the current register-only naive form), GPU elementwise/reduction/softmax/layernorm, and ultimately `wmma` Tensor-Core GEMM — each validated against the CPU `tensor.hx`/`nn.hx` oracle, then GPU autodiff (criterion #4). That is the bulk of the road to the capstone. But the road is now **open**: Helix can drive the GPU.
