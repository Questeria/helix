# Using `kovc`: compile and run Helix programs

*What this chapter covers:* how to take a `.hx` source file, compile it to a Linux ELF with the
`kovc` compiler, and run it; the program entry point and the exit-code convention every Helix
program follows; where the `kovc` binary actually comes from; and a first-program walkthrough using
a real, gate-verified example — quoted verbatim, compiled, and run.

This chapter assumes you have already built the toolchain. If you have not, read
[part2-setup-build/02-build-from-raw.md](02-build-from-raw.md) first — building from the raw root is
what *produces* the `kovc` binary this chapter drives.

---

## The two binaries: `seed` and `kovc`

Helix has two compilers, and it helps to be precise about which one you are using.

- **`seed`** — the Apache-2.0 **C-subset** compiler (`stage0/helixc-bootstrap/`, source `seed.c`),
  built by the from-raw ladder. It is small enough to be the trust bridge from the raw root to
  Helix, and its only job is to compile `kovc`'s Helix source into a native binary.
- **`kovc`** — the Helix compiler **written in Helix** (`helixc/bootstrap/{lexer,parser,kovc}.hx`),
  which lexes, parses, and emits an x86-64 Linux ELF directly — no external assembler, linker, or
  libc. This is the compiler you use to compile your own programs.

Both produce a self-contained ELF you then run on Linux (or WSL2 on Windows). They differ in their
command-line shape, and that difference matters when you script them — so it gets its own section
below.

> **For AI agents:** `seed` and `kovc` are *different binaries with different invocation
> conventions*. `seed` takes file arguments on `argv`; the built `kovc` (`K1`/`K2`) reads and writes
> **fixed `/tmp` paths**. Do not assume one's calling convention for the other. Both facts are
> grounded in real scripts and quoted verbatim below.

## Where the `kovc` binary comes from

You do not download `kovc`, and there is no trusted pre-built compiler anywhere in the chain. `kovc`
is *built*, from the raw root up, and the binary you run is one of the rungs of the self-host
fixpoint:

```text
seed  --compiles-->  K1  --compiles-->  K2  ( == K3 == K4, byte-identical )
```

Concretely, from [`scripts/feature_corpus.sh`](../../../scripts/feature_corpus.sh) (the real build
sequence, quoted verbatim):

```bash
echo "=== build K2 (general full-language compiler) from the raw seed ==="
bash assemble_k1.sh >/dev/null 2>&1
t0=$SECONDS
timeout 400 ./seed.bin k1src.hx /tmp/K1.bin; echo "  seed->K1 rc=$? ($((SECONDS-t0))s)"
chmod +x /tmp/K1.bin
cp k1input.hx /tmp/k1_in.hx
timeout 60 /tmp/K1.bin
[ -s /tmp/k1_out.bin ] || { echo "FATAL: K2 not built"; exit 8; }
cp /tmp/k1_out.bin /tmp/K2.bin; chmod +x /tmp/K2.bin
echo "  K2 built: $(stat -c%s /tmp/K2.bin) bytes"
```

Reading that top to bottom:

1. `bash assemble_k1.sh` regenerates the compiler's *source* — `k1src.hx`, `k1input.hx`, and
   `k1ptxdrv.hx` — by concatenating `lexer.hx` + `parser.hx` + the body of `kovc.hx` with a small
   driver `main`. The assembly logic is itself a Helix program (`assemble_k1.hx`) compiled and run
   by the frozen `seed`; see [`stage0/helixc-bootstrap/assemble_k1.sh`](../../../stage0/helixc-bootstrap/assemble_k1.sh).
2. `./seed.bin k1src.hx /tmp/K1.bin` — the `seed` compiles the assembled Helix compiler source to
   the first native `kovc` binary, **K1**.
3. `cp k1input.hx /tmp/k1_in.hx` then `/tmp/K1.bin` — **K1** compiles the compiler source *again*
   (its input is the same compiler), producing the next stage, which is copied to `/tmp/K2.bin`.

`/tmp/K2.bin` is the `kovc` binary you compile programs with for the rest of this chapter. (The
fixpoint continues to K3 and K4 to *prove* `K2 == K3 == K4` byte-for-byte; that proof is the gate's
job, covered in [part2-setup-build/04-reproduce-verify-trust.md](04-reproduce-verify-trust.md). For
*using* the compiler, K2 is enough.)

> **For AI agents:** K1 and K2 are the **same compiler** at successive fixpoint stages; either
> compiles arbitrary `.hx`. Use the freshly-built `/tmp/K2.bin`. Do not hand-edit `k1src.hx` /
> `k1input.hx` — they are generated by `assemble_k1.sh` from `helixc/bootstrap/*.hx`; edit the
> sources and re-assemble.

## How you invoke `kovc`

Here is the one fact that surprises people: **the built `kovc` (`K1`/`K2`) does not take filename
arguments**. It reads its source from a *fixed* input path and writes the ELF to a *fixed* output
path, both hard-coded into the compiler's driver `main`. This is deliberate — `kovc` is a from-raw,
minimal compiler, and a fixed-path I/O contract is the least machinery that lets it self-host.

The driver that becomes **K2** is [`stage0/helixc-bootstrap/drivers/driver_k1input.hx`](../../../stage0/helixc-bootstrap/drivers/driver_k1input.hx),
quoted in full:

```helix
fn main() -> i32 {
    let src_start = __arena_len();
    let src_len = read_file_to_arena("/tmp/k2_in.hx");
    let tok_base = __arena_len();
    lex(src_start, src_len);
    let ast_root = parse_top(tok_base);
    // H-3: compile-time file:line:col diagnostic on a parse error.
    // Clean input (the self-host source) hits no AST_ERR, so
    // err_off < 0 and the normal emit path runs byte-identically --
    // the self-host fixpoint K2==K3==K4 is preserved.
    let err_off = find_first_err_offset(ast_root);
    if err_off >= 0 {
        print_str("/tmp/k2_in.hx");
        report_parse_diag(src_start, err_off);
        1
    } else {
        let total = emit_elf_for_ast_to_path(ast_root);
        let elf_start = __arena_len() - total;
        write_file_to_arena("/tmp/k2_out.bin", elf_start, total)
    }
}
```

So, to compile a program with **K2**:

- Put your source at **`/tmp/k2_in.hx`** (the literal path the K2 driver reads).
- Run `/tmp/K2.bin`.
- The emitted ELF appears at **`/tmp/k2_out.bin`** (the literal path the K2 driver writes).

The **K1** driver is identical except its fixed paths are `/tmp/k1_in.hx` and `/tmp/k1_out.bin`
(see [`drivers/driver_k1src.hx`](../../../stage0/helixc-bootstrap/drivers/driver_k1src.hx)).

The canonical compile-and-run sequence is the `check()` function in
[`scripts/feature_corpus.sh`](../../../scripts/feature_corpus.sh), quoted verbatim:

```bash
check() { # <abs-file> <expected-exit> <feature-label>
  local f="$1" exp="$2" feat="$3" b; b=$(basename "$1")
  if [ ! -f "$f" ]; then echo "  MISSING       $b  [$feat]"; fail=$((fail+1)); return; fi
  cp "$f" /tmp/k2_in.hx
  rm -f /tmp/k2_out.bin
  timeout 30 /tmp/K2.bin >/dev/null 2>&1
  if [ ! -s /tmp/k2_out.bin ]; then echo "  COMPILE-FAIL  $b  [$feat]"; fail=$((fail+1)); return; fi
  chmod +x /tmp/k2_out.bin
  timeout 10 /tmp/k2_out.bin; local rc=$?
  if [ "$rc" = "$exp" ]; then echo "  PASS          $b  exit=$rc  [$feat]"; pass=$((pass+1));
  else echo "  RUN-FAIL      $b  exit=$rc expect=$exp  [$feat]"; fail=$((fail+1)); fi
}
```

Three details in that function are load-bearing, and they are the difference between scripting
`kovc` correctly and getting fooled by a stale file:

1. **`rm -f /tmp/k2_out.bin` *before* the compile.** The output path is fixed and reused across
   every compile, so you must delete a previous run's ELF first — otherwise a *failed* compile
   leaves the *old* binary in place and you run the wrong program.
2. **Success is judged by `[ -s /tmp/k2_out.bin ]` — a non-empty output file — not by `kovc`'s exit
   code.** `kovc` returns the **number of bytes it wrote** as its process exit status (mod 256), so
   a *successful* compile exits *non-zero*. `feature_corpus.sh` documents this in its header:
   *"K2 = general compiler via fixed paths: stage `.hx` at `/tmp/k2_in.hx`, run K2, read the emitted
   ELF at `/tmp/k2_out.bin`."* And [`scripts/gate_kovc.sh`](../../../scripts/gate_kovc.sh) spells out
   the rule explicitly: *"kovc returns its OUTPUT BYTE COUNT as the process exit status (rc = size
   mod 256 … i.e. NONZERO ON SUCCESS) … those legs are validated by non-empty output … never by rc."*
3. **`chmod +x /tmp/k2_out.bin` before running.** The emitted file is a complete ELF, but it is
   written as a plain file; mark it executable, then run it. *Its* exit code is your program's
   result.

> **For AI agents:** never test `kovc`'s success with `[ $? -eq 0 ]` — a good compile exits non-zero
> (the byte count). Test `[ -s /tmp/k2_out.bin ]` instead, and always `rm -f /tmp/k2_out.bin` before
> the compile so a prior ELF cannot masquerade as this one. These two rules are quoted from
> `feature_corpus.sh` / `gate_kovc.sh`; follow them verbatim.

### For comparison: invoking `seed`

The `seed` compiler *does* take file arguments, in case you need to compile a program with it (for
example, when bisecting the trust chain). From [`stage0/helixc-bootstrap/run_tests.sh`](../../../stage0/helixc-bootstrap/run_tests.sh):

```bash
#   no args  -> seed runs its lexer + parser self-tests (exit 42).
#   in out   -> seed compiles a .hx file to a self-contained ELF.
```

So `./seed.bin <source.hx> <output.bin>` compiles, and `./seed.bin` with no arguments runs `seed`'s
own self-tests and exits 42. This `argv` interface is `seed`'s, **not** `kovc`'s — do not transfer
it.

## The entry point and the exit-code convention

Every runnable Helix program has the same entry point and the same way of reporting a result.

**Entry point.** Execution begins at:

```helix
fn main() -> i32 {
    // ...
}
```

`main` takes no parameters and returns an `i32`.

**Exit code.** The `i32` that `main` returns becomes the process's **exit status**, which on Linux
is an **8-bit** value — so the result is taken **modulo 256**. This is not a Helix quirk; it is the
operating system's exit-status width, and every example file states it the same way. Three
independent confirmations from real sources:

- [`helixc/examples/hbs_sample_recursion.hx`](../../../helixc/examples/hbs_sample_recursion.hx)
  computes `factorial(5)` and notes: *"5! = 120, but exit code is 8-bit → 120 fits."*
- [`helixc/examples/hbs_sample_enum_struct.hx`](../../../helixc/examples/hbs_sample_enum_struct.hx)
  sums to 129 and notes the total is *"Truncated to i32 mod 256 by the OS exit code wrapper, so
  129."*
- [`stage0/helixc-bootstrap/run_tests.sh`](../../../stage0/helixc-bootstrap/run_tests.sh) asserts a
  factorial program's result with the comment *"7! = 5040, exit code = 5040 & 255 = 176."*

The practical consequence: a Helix example that wants to be checkable by its exit code keeps its
result **under 256**. That is exactly why so many of the corpus programs are engineered to return
**42** — a small, unmistakable success sentinel that survives the 8-bit truncation. If your program
naturally produces a value ≥ 256, the exit code will be that value mod 256; print the full value
instead (Helix has `print_int` / `print_str`, used by `kovc`'s own diagnostics) if you need to see
it whole.

> **For AI agents:** to assert a program's result via its exit code, ensure the expected value is in
> `0..=255`; otherwise compare against `value & 255`. The gate and `feature_corpus.sh` do exactly
> this (e.g. `chk "$EX/matmul_2x2.hx" 69`, `chk "$EX/exit42.hx" 42`).

## First program: `exit42.hx`

The canonical first Helix program is [`helixc/examples/exit42.hx`](../../../helixc/examples/exit42.hx).
It is the minimal end-to-end proof: a function, a return, an integer literal — compiled to a real
ELF that runs and exits 42. Here it is, verbatim:

**Verified example** — `helixc/examples/exit42.hx` (compiles to a Linux ELF; runs and exits `42`):

```helix
// First end-to-end Helix program: compiles, links, runs, exits with status 42.
//
// Demonstrates: function decl, return, integer literal.
// Verification: $? == 42 after running the produced ELF.

fn main() -> i32 {
    42
}
```

Two things to notice in the language itself:

- There is no `return` keyword here. The **last expression** of a block is its value, so `42` *is*
  what `main` returns. (Helix has `return` for early exits — see
  [`helixc/examples/hbs_sample_recursion.hx`](../../../helixc/examples/hbs_sample_recursion.hx) and
  the language chapters in Part III — but a tail expression needs no `return`.)
- `42` is an `i32` literal, matching `main`'s `-> i32`.

### Compile it and run it

Using the `/tmp/K2.bin` you built earlier, follow the same steps `feature_corpus.sh` uses. From the
bootstrap directory (`stage0/helixc-bootstrap`), with `$EX` pointing at the examples directory:

```bash
cp "$EX/exit42.hx" /tmp/k2_in.hx   # stage the source at the fixed input path
rm -f /tmp/k2_out.bin              # clear any previous ELF (the output path is reused)
timeout 30 /tmp/K2.bin             # run kovc: reads /tmp/k2_in.hx, writes /tmp/k2_out.bin
chmod +x /tmp/k2_out.bin           # mark the emitted ELF executable
/tmp/k2_out.bin; echo "exit=$?"    # run YOUR program; print its exit code
```

The expected result:

```text
exit=42
```

That `42` is `main`'s return value, surfaced as the process exit status. This exact program/exit-code
pair is one of the gate's baseline rows — [`scripts/gate_kovc.sh`](../../../scripts/gate_kovc.sh)
asserts it with `chk "$EX/exit42.hx" 42` — so if your local run prints anything other than `exit=42`,
something in the build is wrong, not the program.

> **For AI agents:** `exit42.hx → 42` is the canonical smoke test for a working `kovc`. If staging it
> at `/tmp/k2_in.hx`, running `/tmp/K2.bin`, and executing `/tmp/k2_out.bin` does not yield exit `42`,
> stop and re-verify the build (`scripts/gate_kovc.sh`) before trusting any other compile.

## A second program: real arithmetic (`matmul_2x2.hx`)

`exit42.hx` proves the pipeline; the next example proves the compiler computes. `kovc` does the work
at compile time and run time — there is no interpreter in the loop.
[`helixc/examples/matmul_2x2.hx`](../../../helixc/examples/matmul_2x2.hx) multiplies two 2×2 matrices
with scalar `let`s and returns the trace of the product. Verbatim:

**Verified example** — `helixc/examples/matmul_2x2.hx` (compiles to a Linux ELF; runs and exits `69`):

```helix
// matmul_2x2.hx — 2x2 matrix multiplication via scalar ops.
//
// We don't have arrays + indexing yet, so this version operates on individual
// elements. The trace of A*B is computed and returned.
//
//   A = [[1, 2], [3, 4]]
//   B = [[5, 6], [7, 8]]
//   A*B = [[1*5+2*7, 1*6+2*8], [3*5+4*7, 3*6+4*8]]
//       = [[19, 22], [43, 50]]
//   trace(A*B) = 19 + 50 = 69

fn main() -> i32 {
    // We're capped at 3 args per fn in the v0.1 codegen, so call with
    // partial args via a wrapper. Actually let's compute directly inline:
    //   A = [[1, 2], [3, 4]]
    //   B = [[5, 6], [7, 8]]
    let a00 = 1; let a01 = 2; let a10 = 3; let a11 = 4;
    let b00 = 5; let b01 = 6; let b10 = 7; let b11 = 8;
    let c00 = a00 * b00 + a01 * b10;
    let c11 = a10 * b01 + a11 * b11;
    c00 + c11   // 19 + 50 = 69
}
```

Compile and run it with the same five steps, just swapping the file:

```bash
cp "$EX/matmul_2x2.hx" /tmp/k2_in.hx
rm -f /tmp/k2_out.bin
timeout 30 /tmp/K2.bin
chmod +x /tmp/k2_out.bin
/tmp/k2_out.bin; echo "exit=$?"
```

Expected:

```text
exit=69
```

`19 + 50 = 69`, well under 256, so it surfaces cleanly as the exit code — and the gate pins it with
`chk "$EX/matmul_2x2.hx" 69`.

> **Note:** the in-source comments above ("we don't have arrays + indexing yet", "capped at 3 args
> per fn in the v0.1 codegen") describe the *historical* state when this example was written. The
> shipped `kovc` corpus today includes array indexing (`arr_idx`) and functions with many parameters
> (`f8_args`/`f9_args`/`f11_args`), all in [`scripts/gate_kovc.sh`](../../../scripts/gate_kovc.sh).
> The example still compiles and returns 69 exactly as quoted; treat the comments as period notes,
> not current limits.

## When a compile fails

If your source has a parse error, the K2 driver does **not** write an output ELF. Instead it prints a
structured diagnostic and returns 1. The path-printing and the diagnostic are visible in the driver
quoted earlier and in `kovc`'s `report_parse_diag` (`helixc/bootstrap/kovc.hx`): the message form is

```text
/tmp/k2_in.hx:<line>:<col>: parse error: unexpected token
```

Because no ELF is written, the `[ -s /tmp/k2_out.bin ]` check fails — which is exactly how
`feature_corpus.sh`'s `check()` reports `COMPILE-FAIL`. This is the same fail-closed discipline the
trust docs insist on: a compile that did not happen must look like a failure, never silently reuse a
stale binary.

> **Residual:** `kovc` is a from-raw, deliberately-minimal compiler. Its driver reads/writes fixed
> `/tmp` paths and reports the source path as the literal `/tmp/k2_in.hx` (the staged copy), not your
> original filename. There is no `-o`, no flags, no multi-file linking at this layer — that
> machinery is intentionally absent so the compiler stays small enough to audit and self-host. The
> historical Python-hosted `helixc` frontend had a richer CLI, but it is **not** in the shipped
> compile/run path; see [`QUICKSTART.md`](../../../QUICKSTART.md) and the terminology note in the
> [Style Guide](../STYLE_GUIDE.md).

## Putting it together

The whole loop, end to end, is small enough to hold in your head:

1. **Build `kovc` once.** `assemble_k1.sh` → `seed` compiles `k1src.hx` to `K1` → `K1` compiles
   `k1input.hx` to `/tmp/K2.bin`. (See [02-build-from-raw.md](02-build-from-raw.md).)
2. **Stage your source** at `/tmp/k2_in.hx`.
3. **Clear the output** with `rm -f /tmp/k2_out.bin`.
4. **Run `/tmp/K2.bin`** — it reads the fixed input, writes the fixed output ELF. (A non-empty
   `/tmp/k2_out.bin` means success; `kovc`'s own exit code is a byte count, not a status.)
5. **`chmod +x /tmp/k2_out.bin` and run it.** Your `fn main() -> i32` return value is the exit code,
   modulo 256.

Two real programs — `exit42.hx → 42` and `matmul_2x2.hx → 69` — exercise that loop and are pinned by
the gate, so they double as a self-check that your `kovc` works.

If you want to confirm not just *one* program but the whole compiler against its pinned anchors —
the byte-identical self-host fixpoint plus the full feature corpus — that is the gate's job, and the
one-command reproduction is `bash scripts/reproduce_trust.sh` on a clean checkout (it also runs on a
fresh CI runner via [`.github/workflows/trust-reproduce.yml`](../../../.github/workflows/trust-reproduce.yml)).
