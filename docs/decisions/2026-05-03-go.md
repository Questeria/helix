# Decision: Begin Kovostov-Native, raw-binary bootstrap, fully open-source

**Date**: 2026-05-03
**Author**: Claude (autonomous, autonomy 5)
**Status**: committed

## Context

After multi-turn exploration of foundations (binary as floor, Rust vs lower-level hosts, bootstrap problem), the user committed to:
1. Raw binary as the starting point (hard constraint)
2. AI named **Kovostov**
3. Project in its own folder
4. End goal: completely open-source AGI

User said "Go" with full autonomy: I execute the plan, update it as I learn, and may disregard it if better paths emerge.

## Decisions

### D1 — Project location: `C:/Projects/Kovostov-Native/`
Sibling of the existing `C:/Projects/Kovostov/` (the Claude-Code shell framework). Sibling, not subdirectory, because:
- The native build is a different artifact with a different lifecycle
- The shell framework continues to exist as a cognitive scaffold during build
- Once Kovostov-Native is operational, the shell is retired or repurposed

### D2 — Bootstrap from raw binary, stage0 lineage
Hand-encoded hex seed (~150–250 bytes), modeled on stage0 / Live Bootstrap / GNU Mes. Each subsequent stage's binary is produced by the previous stage. Audit-only use of `objdump`, `xxd`, `nasm`, `cmp` for cross-checking — never for producing shipped artifacts.

Alternative considered and rejected: Rust-host bootstrap with retroactive raw-binary chain as a later phase. Rejected because the user set raw-binary as a hard constraint, not a deferred goal.

### D3 — Bootstrap target: Linux x86_64 ELF (via WSL2 on Windows)
ELF is simpler than PE (~64 bytes minimal header vs ~hundreds for PE), syscall ABI is cleaner, the entire stage0 / M2-Planet ecosystem targets Linux. WSL2 on Windows 11 Pro is a 5-minute install.

The eventual `helixrt` runtime targets Windows-native (CUDA Driver API → RTX 5090) because GPU work is materially cleaner there. Bootstrap and AI runtime can be on different OS targets without contradiction.

### D4 — Stack identity
- System / AI: **Kovostov**
- Language: **Helix** (`.hx`)
- Compiler: **helixc**
- Runtime: **helixrt**

### D5 — License
- Code: **Apache 2.0** (modern OSS standard, explicit patent grant)
- Documentation: **CC-BY 4.0** (allows derivatives with attribution)
- Trained model weights: **CC0** (public domain dedication, maximum freedom)

Considered: GPL/AGPL for stronger copyleft. Rejected because:
- The user said "completely open source," and Apache + CC0 is the format that maximizes freedom of *use*, including in commercial derivatives
- Forces no obligation on adopters, maximizing adoption and ecosystem benefit
- Open-source AI weights under restrictive licenses (OpenRAIL, Llama license) have been criticized; CC0 sidesteps this entirely

If the user later prefers AGPL + RAIL, we can re-license; CC0 weights remain public domain regardless.

### D6 — Language design committed (from prior research)
- Spiritual ancestor: Dex + Triton + JAX + Enzyme
- Type system: Futhark size types + opt-in refinements (Presburger)
- Tensor primitive: tile-as-first-class with memory-space tag
- Element type: `(format, block, scale_format)` triple, supports BF16/FP8/FP4/Ternary
- Autodiff: compiler pass on optimized IR, library API
- IR: typed SSA with block parameters

### D7 — Bootstrap chain stages
hex0 → hex1 → M0 → M1 → M2-Planet → helixc-bootstrap → helixc

Each stage's input format is text (initially hex digits with whitespace; later, assembler mnemonics; later, a C subset; finally Helix). Each stage's binary is produced by the previous stage. The whole chain is reproducible from `stage0/hex0.bin` plus the source of each later stage.

### D8 — Phase 0 first artifact
The hex0 seed monitor: reads hex characters from stdin, skips whitespace and `;`-comments, pairs hex digits into bytes, writes bytes to stdout. ~150–250 bytes of hand-encoded x86-64 ELF. Annotated byte-by-byte.

This is the smallest unit of "we wrote it from raw binary." Once it works, the bootstrap chain has begun.

### D9 — No artificial human bottleneck
User has authorized full autonomy and unlimited agent use. I will spawn parallel research / planning / writing agents wherever the work parallelizes, and report periodic status updates.

## Rejected alternatives

- **Rust host first**: rejected per user's explicit raw-binary constraint
- **Adopt M2-Planet's hex0 verbatim**: kept as fallback for month-3 timeline gate; default is to write our own
- **Windows PE bootstrap**: rejected — PE is ~3× more complexity per stage with no benefit at this layer
- **Self-hosting helixc directly without M2 step**: rejected — too large a leap; M2-class C-subset compiler is well-trodden

## Reversibility

All of these are reversible by editing this file and adding a follow-up decision. The bootstrap chain is the most committed-to piece; even there, switching to Rust-host (with full audit chain as a separate later artifact) is reversible at any phase boundary if the timeline blows up.
