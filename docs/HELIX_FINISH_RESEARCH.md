## Path to real GPU execution — THE GATE (Helix v1.0 DoD criterion #3)

**Current state.** VERIFIED ON THIS MACHINE: a pure CUDA Driver-API launcher I wrote (cuInit -> cuDeviceGet -> cuCtxCreate -> cuModuleLoadData(ptx_text) -> cuModuleGetFunction -> cuMemAlloc -> cuMemcpyHtoD -> cuLaunchKernel -> cuCtxSynchronize -> cuMemcpyDtoH) compiled with `gcc launch.c -I/usr/local/cuda/include -L/usr/lib/wsl/lib -lcuda` and EXECUTED a vector_add PTX kernel on the RTX 3070 Laptop GPU, result verified (c[7]=21.0 PASS). So the entire downstream path works today; the ONLY missing link is how Helix drives that launcher.

Environment (all confirmed via WSL2): libcuda.so/.so.1/.so.1.1 at /usr/lib/wsl/lib/libcuda.so (driver 595/596) exporting cuInit/cuDeviceGet/cuCtxCreate/cuModuleLoadData/cuModuleGetFunction/cuMemAlloc/cuMemcpyHtoD/cuMemcpyDtoH/cuLaunchKernel; a -lcuda link stub at /usr/local/cuda-12.8/targets/x86_64-linux/lib/stubs/libcuda.so; nvcc + ptxas (CUDA 12.8, also a 12.0 nvcc) at /usr/bin + /usr/local/cuda/bin; cuda.h driver header present; nvidia-smi sees the RTX 3070 8GB (some telemetry fields show ERR! — a known cosmetic WSL passthrough quirk; compute is unaffected, proven by the successful launch).

Compiler side (helixc/bootstrap/, pure Helix, K4 done — Python deleted):
- PTX emitter EXISTS and is real: emit_ptx_for_ast_to_path (kovc.hx:11440), emit_ptx_entry (kovc.hx:11372), ~800 emit_ptx_* refs; ptxas-validated through matmul; 30 golden tests. nvcc-emitted PTX for vector_add has the IDENTICAL module skeleton the bootstrap emits (.version 8.0 / .target sm_75 / .address_size 64 / .visible .entry name(...)). Output-mode dispatcher emit_auto_for_ast_to_path (kovc.hx:11533) already routes @kernel->PTX vs CPU->ELF.
- FFI/exec mechanisms the bootstrap has TODAY (all syscall-based, x86_64 machine code emitted directly): run_process(path: STRLIT) = fork(57)+execve(59)+wait4(61) returning child exit code (emit_run_process_body kovc.hx:4325, dispatch :5412); set_exec(path) = chmod 0o755 (:4382/:5429); read_file_to_arena / write_file_to_arena (:4077/:4201); mmap (sys_mmap=9) used for the big-stack stub.

This is the core tension: libcuda.so is a DYNAMIC shared library and Helix's "talk directly to the chip / static syscall-only" model cannot link or call it without new machinery.
- run_process LIMITATION: it passes argv=[path, NULL] only — NO arguments. So invoking `nvcc -o x x.cu` or `gcc ... -lcuda` (which need args) requires either extending run_process to accept argv, or shelling through a fixed-path wrapper script written by write_file_to_arena.
- The bootstrap's actual main() (kovc.hx:12171) is still a HARDCODED smoke harness (emits 42 to a fixed path). The real read-source/parse/route/write compiler-driver main is exercised only via the Python test harness (_kovc_self_host_emit_ptx). Standalone Helix driver-main is the in-progress test-infra T1/T2 campaign.

Path (c) nvcc is strictly worse than (a) here (we already have the PTX; nvcc would only re-wrap it and adds a heavier external dep).

5. GPU: defer until criterion #3 (tile execution on real HW) is unblocked; then re-emit the same backward kernels as tile kernels and gradient-check on the RTX 3070. Autodiff-on-GPU is gated by GPU execution, not by AD design.

**Risks:**

- Forward and reverse modes have DIVERGENT op coverage (forward has 6 transcendentals; reverse has none) — any code relying on grad over a transcendental in reverse mode silently traps to ud2.
- Fail-closed-to-0 in forward softmax/layernorm/attention means a NaN during training produces a wrong-but-finite gradient with no signal, which can corrupt convergence invisibly (capstone requires loss within 2% of PyTorch oracle).
- No tensor-level reference oracle exists post-K4 except external PyTorch; verifying tensor-op gradients requires standing up a trusted comparison that is itself outside the Python-free toolchain.
- The scalar grad intrinsic gives a false impression of 'autodiff done' — it cannot differentiate any of the transformer's actual tensor ops (matmul/softmax/layernorm/attention); those gradients are entirely manual and partly missing.

**Effort.** Medium-large; the GPU half sits BEHIND the GPU gate (#3) in dependency order. CPU half is independently actionable now: finite-diff checker = small; each missing backward kernel (matmul/softmax/layernorm/attention) = small-medium and self-contained; forward-mode transcendental chain rules = small (pattern-copy). **Blockers:**

- GPU autodiff is hard-blocked by criterion #3 (tile/PTX emitters are stubs — 'THE GATE'); no gradient can run on the RTX 3070 until real-HW tile execution exists.
- The gradient-check suite needs the Helix test runner (test-infra T2, in progress) to be a Python-free acceptance test (criterion #6 forbids .py in the live toolchain).
- The deleted Python reference (HELIX_STAGE30_COMPILER_SNAPSHOT/) was the only trusted AD-parity reference and was scalar-only; there is currently no tensor-level reference oracle except the external PyTorch oracle named in the capstone.
