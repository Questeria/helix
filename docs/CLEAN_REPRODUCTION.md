**Where it walls (honest portability caveat):** the **fixpoint layer** is path-locked.
`assemble_k1.hx` (which the gate runs at step [0] to regenerate `k1src.hx / k1input.hx /
k1ptxdrv.hx`) **hardcodes absolute `/mnt/c/Projects/Kovostov-Native/...` paths** for both its
reads (lexer/parser/kovc/drivers) and its writes (the k1*.hx outputs) ‚Äî 9 such paths. Running
`assemble_k1.sh` *inside the clone* returns rc=0 but **reads from and writes to the CANONICAL
dir, not the clone** (verified: the k1*.hx never appeared in the clone; the canonical
`k1src.hx` was the file written). On a machine where the canonical dir is **absent**, that
hardcoded `read_file_to_arena("/mnt/c/Projects/Kovostov-Native/...")` would fail and the
gate's step [0] ‚Äî hence the whole self-host fixpoint ‚Äî would not run.

**Caveat for the trust record:** a clean checkout reproduces the **from-raw ladder ‚Üí seed**
(the root of trust, including the gcc-DDC anchor of Step 4, which uses only relative paths and
the frozen `seed.c`) **from any directory**. The **self-host fixpoint + GPU gate + capstone**
currently require the checkout to live at the canonical `/mnt/c/Projects/Kovostov-Native`
path, because `assemble_k1.hx` hardcodes it. This is a build-hygiene portability limitation
(parameterize the path / use a repo-relative base), **not** a trust gap in the chain ‚Äî the
bytes produced are identical; only the *location* the build assumes is fixed. (The clone was
removed after the test; the canonical gitignored k1*.hx are regenerated by every gate run.)
